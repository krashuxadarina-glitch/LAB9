### **Відповіді на контрольні запитання**

1. **Чому в конфігураційних файлах паролі не зберігаються в явному вигляді?** Паролі зберігаються у вигляді криптографічних хешів (зазвичай у файлі `/etc/shadow`), щоб забезпечити конфіденційність. Навіть якщо зловмисник отримає доступ до файлу, він не зможе відразу дізнатися пароль, оскільки хешування — це незворотний процес. Це захищає систему від несанкціонованого доступу у разі витоку даних.                                                                                                 <img width="502" height="122" alt="image" src="https://github.com/user-attachments/assets/f74fbd79-236b-44ee-8f7d-98967338f8ac" />

2. **Чому не рекомендується виконувати повсякденні операції, використовуючи обліковий запис root?** Обліковий запис `root` має необмежені права. Будь-яка помилка в команді (наприклад, друкарська помилка в `rm`) або запуск шкідливого ПЗ від імені суперкористувача може призвести до повного знищення системи або втрати даних без можливості відновлення. Повсякденна робота під звичайним користувачем створює захисний бар'єр.                                                             <img width="738" height="105" alt="image" src="https://github.com/user-attachments/assets/cb579fcc-705e-4716-95ec-a816a4602ee8" />

3. **У чому відмінність механізмів отримання особливих привілеїв su і sudo?**  
   * **su (Switch User):** Повністю перемикає поточну сесію на іншого користувача (за замовчуванням `root`). Для цього потрібно знати пароль саме того користувача, на якого ви перемикаєтесь.  
   * **sudo (SuperUser Do):** Дозволяє виконати конкретну команду з правами адміністратора, використовуючи пароль **поточного** користувача. Це безпечніше, оскільки не потребує розголошення пароля `root` і дозволяє гнучко налаштовувати права доступу через файл `/etc/sudoers`.                       <img width="732" height="229" alt="image" src="https://github.com/user-attachments/assets/9cf5280a-4596-437e-bbeb-2b80d3adc724" />

4. **Чому домашній каталог користувача root не розміщено в каталозі /home?** Каталог `/root` знаходиться в кореневому розділі для того, щоб адміністратор міг увійти в систему та провести відновлення навіть у випадку, якщо розділ `/home` (який часто монтується окремо) пошкоджений або не зміг завантажитися.                                                                                <img width="614" height="121" alt="image" src="https://github.com/user-attachments/assets/c1b306ac-cb30-4e0a-8fe4-067091765bce" />

5. **Для чого використовується команда getent?** Команда `getent` використовується для отримання записів з адміністративних баз даних (таких як `passwd`, `group`, `hosts`). Її головна перевага над простим читанням файлів (наприклад, `cat /etc/passwd`) у тому, що вона відображає користувачів не лише з локальних файлів, а й з мережевих служб (LDAP, AD), якщо вони підключені.                   <img width="622" height="69" alt="image" src="https://github.com/user-attachments/assets/91058ad4-81bb-43bb-b893-670db0b3d27a" />

6. **Як можна змінити пароль користувача?** Для цього використовується команда `passwd`. Якщо її ввести без аргументів, вона змінить пароль поточного користувача. Адміністратор може змінити пароль будь-якого іншого користувача командою `sudo passwd [ім'я_користувача]`.                             <img width="726" height="380" alt="image" src="https://github.com/user-attachments/assets/93163c1f-a753-4736-a01e-08cc9db48577" />

7. **Яким чином можна видалити існуючі групи користувачів? Чи залишиться інформація про них десь у системі?** Групи видаляються командою `groupdel`. Після виконання команди запис про групу видаляється з файлів `/etc/group` та `/etc/gshadow`. Проте файли, що належали цій групі, залишаться в системі, але замість імені групи в їхніх атрибутах буде відображатися лише числовий ідентифікатор (GID), який раніше належав цій групі.                                                                   <img width="695" height="171" alt="image" src="https://github.com/user-attachments/assets/4734e03f-77c6-4c51-b665-a3056f5f3d9d" />

8. **Яке призначення команди chage?** Команда `chage` (Change Age) використовується для керування політикою застарівання паролів. Вона дозволяє встановити дату закінчення терміну дії пароля, мінімальну та максимальну кількість днів між змінами пароля, а також налаштувати попередження про необхідність зміни.                                                                                  <img width="656" height="187" alt="image" src="https://github.com/user-attachments/assets/a7d7043f-6d35-4807-9652-fc4446a99d59" />

9. **Які параметри команди usermod ви вважаєте найбільш використовуваними?**  
   * `-aG`: Додавання користувача до додаткових груп без видалення з існуючих.  
   * `-l`: Зміна логіна (імені користувача).  
   * `-L` / `-U`: Блокування та розблокування облікового запису.  
   * `-d -m`: Зміна домашнього каталогу з перенесенням вмісту.  
   * `-s`: Зміна командної оболонки (наприклад, з `/bin/sh` на `/bin/bash`).                             <img width="500" height="130" alt="image" src="https://github.com/user-attachments/assets/216cd43e-65e1-48fd-82ba-94f9e12deed0" />




Завдання для попередньої підготовки:

1\. Поняття UPG (User Private Groups)

UPG (Користувацька приватна група) — це схема керування користувачами, за якої для кожного нового користувача система автоматично створює окрему групу з таким самим ім'ям, як і логін користувача. Цей користувач стає єдиним членом цієї групи.

Коли їх доцільно використовувати?

* Стандарт безпеки: Це дозволяє уникнути ситуації, коли всі користувачі мають одну спільну групу (наприклад, users) за замовчуванням. У схемі UPG права доступу за замовчуванням (umask 002 або 022\) роблять домашню директорію користувача безпечнішою.  
* Спільна робота (SetGID): Це ідеально підходить для проектної роботи. Коли кілька користувачів працюють у спільній папці, UPG дозволяє безпечно налаштовувати права, не впливаючи на інших "сусідів" по системі.

2\. Команди для створення груп

Основним інструментом є команда groupadd.

* Звичайна група:  
  sudo groupadd developers — створює групу з наступним вільним GID (\>= 1000).  
* Системна група:  
  sudo groupadd \-r security\_audit — створює групу з GID у системному діапазоні (1–999).  
* Група з конкретним ID:  
  sudo groupadd \-g 1500 project\_x — створює групу з чітко вказаним ідентифікатором 1500\.

<img width="662" height="107" alt="image" src="https://github.com/user-attachments/assets/9d3896f1-0bbf-45d5-b5de-7e5447541651" />

3\. Команди для зміни налаштувань груп

Для цього використовується команда groupmod, а для керування складом групи — gpasswd або usermod.

* Зміна назви групи:  
  sudo groupmod \-n managers sales — перейменовує групу sales на managers.  
* Зміна GID групи:  
  sudo groupmod \-g 3000 managers — змінює цифровий ідентифікатор групи.  
* Додавання користувача до групи:  
  sudo usermod \-aG managers student — додає користувача student до додаткової групи managers.  
* Видалення групи:  
  sudo groupdel managers — видаляє групу (якщо вона не є основною для жодного користувача).

Glossary of Linux Administration Terms (Словник термінів)

| Term | Definition |
| :---- | :---- |
| UID (User Identifier) | A unique numerical ID assigned to each user in a Linux system. |
| GID (Group Identifier) | A unique numerical ID assigned to each group. |
| UPG (User Private Group) | A security scheme where a unique group is created for every new user. |
| Primary Group | The main group a user belongs to (usually matches the username in UPG). |
| Supplementary Group | Additional groups a user can join to gain specific permissions (Secondary groups). |
| Shell | A program that interprets commands typed by the user (e.g., /bin/bash). |
| Home Directory | A private storage space for a user's files and configurations. |
| Sudo (Superuser Do) | A command that allows users to run programs with the security privileges of the root user. |
| Hashed Password | A cryptographic representation of a password stored in /etc/shadow. |
| Load Average | A measure of the amount of computational work that a computer system performs. |
| Uptime | The duration for which a system has been continuously running. |
